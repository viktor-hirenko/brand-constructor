# Brand Constructor — Анализ флоу и коммуникаций

> **Дата анализа:** 19 марта 2026  
> **Цель:** Полная ревизия процесса взаимодействия команд, уведомлений и того, что ещё не реализовано

---

## 1. ОПИСАНИЕ ТЕКУЩЕГО ФЛОУ (как задумано в PRD)

### 1.1 Общая схема (General Flow из FigJam)

```
Order New Brand → Brand Basics → Mode → Concept → External Naming → Internal Naming 
→ Brand Preview → Marketing Package → Deliverables → Visual Components 
→ All Information View → CEO Approval?
   ├─ NO → Edit Info (возврат на Brand Basics)
   └─ YES → In work → "відправка кожної частини по командам"
```

### 1.2 Распределение данных по командам (из sticky note в FigJam)

| Команда | Что получает |
|---------|--------------|
| **Brand Strategy & Identity** | Brand Basics, Mode, Concept, External Naming, Internal Naming, Brand Preview |
| **PR Team** | Brand Basics, Marketing Package |
| **Service Product Design** | Brand Basics, Mode, Deliverables, Visual Components |

### 1.3 Роли и их точки входа

| Роль | Где работает | Что делает |
|------|--------------|------------|
| **Product Owner** | **Конструктор** | Заполняет бриф (шаги 1–10), отправляет на согласование |
| **CEO (admin, head_dhc, cpo_ceo)** | **Конструктор** (Step 10) | Утверждает или возвращает на доработку |
| **Strategy & Identity** | **Админка** | Добавляет концепты, неймінги в библиотеку |
| **UI Designer** | **Админка** | Добавляет UI-компоненты в библиотеку |
| **PR & Marketing** | **Админка** | Редактирует PR-пакеты в библиотеке |
| **Product Designer** | Только просмотр | — |

## 2. КУДА ВЕДЁТ КЛИК: Конструктор vs Админка

### 2.1 Текущая реализация

**Клик по бренду в админке (BrandsView):**
```javascript
window.open(`${VITE_CONSTRUCTOR_URL}/constructor/brand/${brand.id}`, '_blank')
```
→ Открывается **конструктор** с этим брендом (Step 10 — финальный обзор).

### 2.2 Логика по сценариям

| Сценарий | Куда идёт пользователь | Зачем |
|----------|------------------------|-------|
| Product Owner создаёт новый бриф | Конструктор `/constructor/step/1` | Заполнить бриф |
| Product Owner дорабатывает после "Needs Revision" | Конструктор `/constructor/brand/:id` | Редактировать и отправить снова |
| CEO проверяет и утверждает | Админка → Brands → клик по бренду → Конструктор | Ревью в Step 10 |
| Strategy & Identity добавляет концепт | **Админка** → Concepts | Наполнение библиотеки |
| UI Designer добавляет компонент | **Админка** → Components | Наполнение библиотеки |
| PR добавляет/редактирует пакет | **Админка** → PR Packages | Наполнение библиотеки |

**Вывод:** По флоу в **админку** заходят только для работы с библиотеками (концепты, неймінги, компоненты, PR-пакеты). Для работы с конкретным брендом (создание, доработка, ревью) используется **конструктор**.

## 3. ЧТО ДОЛЖНО ПРОИСХОДИТЬ ПРИ УВЕДОМЛЕНИЯХ (по PRD)

### 3.1 Триггеры уведомлений (из PRD v2, Этап 3: Export)

1. **После утверждения CEO** — бриф уходит командам:
   - Confluence: страница "New Brand #N"
   - Jira: задачи для каждой команды
   - Slack: сообщения в каналы с кратким описанием и ссылками

2. **Ожидаемое содержимое уведомления (Slack/Email):**
   - Краткое описание бренда
   - Ссылка на Confluence (или PDF-бриф)
   - Ссылка на Jira-задачи (если есть)
   - Возможно: прямая ссылка на конструктор для просмотра

### 3.2 Куда ведёт клик по уведомлению (по замыслу PRD)

| Получатель | Действие после уведомления | Куда ведёт ссылка |
|------------|----------------------------|-------------------|
| Brand Strategy & Identity | Получить свой блок брифа | Confluence / PDF / (опционально) конструктор для просмотра |
| PR & Marketing | Получить свой блок | Confluence / PDF |
| Service Product Design | Получить свой блок | Confluence / PDF |
| Product Owner (при "Needs Revision") | Доработать бриф | **Конструктор** `/constructor/brand/:id` |
| CEO | Проверить новый submitted бриф | Админка → Brands → клик → **Конструктор** |

**Важно:** Команды (Strategy, PR, Design) **не редактируют** бриф в конструкторе. Они получают информацию и работают в своих инструментах (Figma, Jira, Confluence и т.д.). Конструктор для них — источник данных, а не рабочее место.

## 4. ЧТО НЕ РЕАЛИЗОВАНО (сводка)

### 4.1 Критичные пробелы (блокируют полный флоу)

| # | Компонент | Статус | Влияние |
|---|-----------|--------|---------|
| 1 | **Slack-уведомления** | ❌ Не реализовано | Никто не узнаёт о submitted/approved/revision |
| 2 | **Email-уведомления** | ❌ Не реализовано | Альтернатива Slack отсутствует |
| 3 | **Confluence API** | ❌ Не реализовано | Нет страницы брифа, некуда вести ссылку |
| 4 | **Jira API** | ❌ Не реализовано | Нет задач для команд |

### 4.2 Текущее состояние коммуникации

- **Product Owner** отправляет бриф → статус `submitted`. Дальше CEO узнаёт о нём только если зайдёт в админку и посмотрит список брендов.
- **CEO** возвращает на доработку → статус `needs_revision`. Product Owner не получает уведомления.
- **CEO** утверждает → статус `approved`. Команды не получают уведомлений и не видят брифа в Confluence/Jira.

### 4.3 Дополнительные пробелы (из аудита PRD)

- Галерея 2–3 изображений для концепта
- iPhone mockup preview на Step 9
- Полноценный PDF-экспорт (сейчас только `window.print()`)
- Форматированный ID брифа (BRIEF-2026-001)
- Бейдж "Used in brand" для PR-пакетов в админке

## 5. РЕКОМЕНДУЕМЫЙ ФЛОУ УВЕДОМЛЕНИЙ (что реализовать)

### 5.1 Минимальный вариант (без Confluence/Jira)

**Триггеры:**
1. Бриф `submitted` → уведомление CEO (Slack или Email)
2. CEO вернул `needs_revision` → уведомление Product Owner (Slack или Email) + ссылка на конструктор
3. CEO `approved` → уведомление трём командам (Slack или Email) + ссылка на PDF или конструктор (read-only)

**Содержимое уведомления:**
- Текст: "Бриф [Internal Name] отправлен на согласование" / "Бриф [X] возвращён на доработку" / "Бриф [X] утверждён"
- Кнопка/ссылка: `{CONSTRUCTOR_URL}/constructor/brand/{id}` (для Product Owner — редактирование, для остальных — просмотр)

### 5.2 Полный вариант (с Confluence/Jira)

- После `approved`: создаётся страница Confluence, задачи в Jira
- В Slack/Email: ссылки на Confluence и Jira вместо/вместе со ссылкой на конструктор

### 5.3 Email как запасной канал

Если Slack API недоступен, можно использовать Email:
- Resend, SendGrid, Mailgun и т.п.
- Шаблоны писем с кнопкой "Открыть бриф" → ссылка на конструктор
