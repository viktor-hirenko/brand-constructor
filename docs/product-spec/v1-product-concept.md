## 1. Executive Summary

### Overview

New Brand Constructor – це внутрішня адмін-панель у вигляді інтерактивного опитувальника, яка централізує процес створення нових брендів онлайн казино та скорочує час запуску продукту на 40-50%.

### Current Challenge

Команда створює 2-4 нових бренди онлайн казино щороку. Зараз кожна команда (Strategy & Identity, PR & Marketing, Product Design) окремо комунікує з Product Owner для з'ясування вимог, що призводить до:

- Фрагментованої комунікації
- Дублювання питань
- Збільшення часу на координацію
- Ризику втрати інформації

### Solution Value

- **Час**: Скорочення часу брифування з ~2 тижнів до 2-3 днів
- **Ефективність**: Всі команди отримують інформацію одночасно
- **Якість**: Використання готових концептів з бібліотеки замість створення з нуля
- **Прозорість**: Єдине джерело правди для всіх stakeholders

## 2. Current State Analysis

### Teams & Processes

#### Strategy & Identity Team

- **Склад**: Brand Strategist, Art Director, Brand Designer
- **Поточний процес**:
  - Проводять брифування з Product Owner
  - З'ясовують ГЕО та стильові побажання
  - Створюють концепти та неймінги
  - Презентують варіанти
- **Проблеми**:
  - Брифування займає 3-5 зустрічей
  - Часто концепти створюються з нуля, навіть якщо є готові рішення
  - Немає централізованої бібліотеки минулих концептів

#### PR & Marketing Team

- **Поточний процес**:
  - Визначають маркетинговий пакет для запуску (4 варіанти)
  - Узгоджують чи потрібен запуск соціальних мереж
  - Планують комунікаційну стратегію
- **Проблеми**:
  - Узгодження відбувається окремо від інших команд
  - Часто питання дублюються

#### Product Design Team

- **Поточний процес**:
  - Визначають чи потрібен лендінг для юристів (отримання ліцензії)
  - З'ясовують необхідність стартового продукту для партнерів
  - Узгоджують візуальний набір компонентів (банери, таббар, хедер, сабнейли, сайдбар, тема)
- **Проблеми**:
  - Product Owner не бачить результат до фінальної презентації
  - Немає можливості швидко протестувати різні комбінації компонентів

#### Approval Chain

Product Owner → CPO → CEO

**Проблема**: Якщо CEO не затверджує концепт, весь процес повторюється заново.

## 3. Solution Overview

### Concept

New Brand Constructor – це опитувальник з візуальними рішеннями, який Product Owner заповнює ПЕРЕД початком роботи над новим брендом. Система інтегрує бібліотеку готових концептів, неймінгів та візуальних компонентів.

### Key Features

#### 1. Бібліотека готових рішень

- Концепти (назва + опис + візуал + лого + неймінги + preview)
- Зовнішні неймінги (окремі або прив'язані до концептів)
- Внутрішні неймінги
- Візуальні варіанти компонентів UI

#### 2. Live Preview

Статичний шаблон мобільного екрану, який оновлюється в реальному часі при виборі:

- Концепту (відображаються банери з візуалом концепту)
- Неймінгу (відображається назва на екранах)
- Компонентів (відображається обраний варіант UI)

#### 3. Centralized Output

Після заповнення автоматично:

- Створюється Confluence page "New Brand #N" з усіма відповідями та візуалом (треба обговорити)
- Надсилаються нотифікації в Slack (треба обговорити)
- (Опціонально) Створюються епіки в Jira для кожної команди (треба обговорити)

## 5. Library Management System

### Content Structure

#### Концепти

**Повний концепт містить:**

- Назва концепту (наприклад, "WonderLand")
- Опис концепту (тематика, настрій)
- Візуальне рішення (бинери з іконками стилістичними)
- 2-3 неймінги для концепту
- Preview на шаблоні казино (банери підставлені під концепт)

**Частковий концепт:**

- Може бути без неймінгів (неймінги додаються окремо пізніше)

#### Неймінги

**Зовнішній неймінг:**

- Може бути прив'язаний до концепту
- Може бути окремим (не прив'язаний до жодного концепту)

**Внутрішній неймінг (для команди):**

- Завжди окремий
- Не прив'язується до концептів

#### Компоненти UI

**Типи компонентів (MVP):**

- Банери
- Таббар
- Хедер
- Сабнейли (thumbnails)
- Сайдбар
- Тема (світла/темна)

Кожен компонент має:

- Кілька варіантів UI (Тип 1, Тип 2, Тип 3...)
- Однакову логіку, різний візуал

### Library Administration

#### Додавання елементів

**Хто може додавати:**

- Admin: всі типи контенту
- Strategy & Identity (Art Director, Strategist): концепти, неймінги
- UI Designers: компоненти та їх варіанти

**Процес додавання концепту:**

1. Заповнення форми (назва, опис)
2. Завантаження візуалу
3. Додавання 2-3 неймінгів (опціонально)
4. Генерація preview на шаблоні
5. Публікація в бібліотеку

**Процес додавання неймінгу:**

1. Вибір типу (зовнішній/внутрішній)
2. Введення назви
3. Опціонально: прив'язка до концепту
4. Публікація

**Процес додавання компонента:**

1. Вибір типу компонента (банер, таббар тощо)
2. Завантаження варіантів UI
3. Публікація

#### Архівування та видалення

**Логіка архівування:**

- Якщо елемент використаний у New Brand #N:
  - Статус змінюється на "Used in Brand #N"
  - Елемент автоматично архівується
  - Не показується в конструкторі при виборі
  - Залишається видимим у створеному бренді
  - Не може бути видалений повністю
- Якщо елемент не використаний:
  - Може бути видалений повністю
  - Запит підтвердження перед видаленням

**Історія використання:**

Кожен елемент має лог:

- Дата створення
- Автор
- Дата використання (якщо застосований)
- В якому бренді використаний

## 6. Product Owner Flow (Constructor Steps)

### Step 1: Brand Basics

**Input:**

- Radiobutton InHouse/WhiteLabel
- Internal project name (auto-generates "New Brand #N")
- Target GEOs / checkbox – New Geo
- Brief description / goals

**Output saved:** Brand ID, GEOs, research – y/n, description

### Step 2: Concept Selection

**Interface:**

- Grid view бібліотеки концептів
- Кожна картка показує:
  - Preview image
  - Назва концепту
  - Короткий опис
  - Індикатор: повний/частковий концепт

**Interaction:**

- Клік на концепт → розгорнутий view:
  - Детальний опис
  - Мудборд
  - Прив'язані неймінги (якщо є)
  - Preview на шаблоні казино
- Або Create New Concept >> інша гілка з брифом від Strategy & Identity Team, далі обирається PR пакет. Все інше формується під Create New Concept з полем для опису.

**Selection:**

- Radio button для вибору одного концепту
- Текстове поле для коментарів (опціонально): "Підходить, але треба подумати над назвою"

**Live Preview Update:**

- Банери (якщо були підвантажені до концепту) в мобільному шаблоні справа відображають обраний концепт

**Output saved:** Concept ID, comments (if any)

### Step 3: External Naming

**Interface:**

- Dropdown або grid view неймінгів
- Фільтри:
  - Всі неймінги
  - Тільки прив'язані до обраного концепту
  - Окремі неймінги
- Кожна опція показує:
  - Назву
  - Тип (прив'язаний до концепту / окремий)

**Selection:**

- Radio button для вибору
- Текстове поле для коментарів

**Live Preview Update:**

- Назва відображається на хедері та інших екранах шаблону

**Output saved:** External naming ID, comments (if any)

### Step 4: Internal Naming

**Interface:**

- Dropdown список внутрішніх неймінгів
- Або можливість ввести новий

**Selection:**

- Radio button для вибору з існуючих
- Або текстове поле для нового неймінгу
- Поле для коментарів

**Output saved:** Internal naming, comments (if any)

### Step 5: Marketing Package

**Interface:**

- 4 картки пакетів (InHouse):
  - Package 1
  - Package 2
  - Package 3
  - Package 4
- Кожна картка показує:
  - Що входить в пакет
  - Задіяні команди
  - Опис пакета
  - Цілі пакета
  - Складові пакета
  - Таймлайн запуску
  - Витрати

**Selection:**

- Radio button для вибору одного пакету
- Поле для коментарів

**Output saved:** Package ID, comments (if any)

### Step 6: Legal Landing

**Question:** Чи потрібен лендінг для юристів (отримання ліцензії)?

**Interface:**

- Toggle або checkbox: Yes / No
- Пояснення: "Лендінг потрібен для отримання ліцензії в регуляторних органах"

**Output saved:** Boolean (true/false)

### Step 7: Partner Starter Product

**Question:** Чи потрібен стартовий продукт для партнерів?

**Interface:**

- Toggle або checkbox: Yes / No
- Пояснення: "Шаблонний тестовий продукт для запуску партнерів (платіжки, провайдери) і підписання договорів"

**Output saved:** Boolean (true/false)

### Step 8: Visual Components Assembly

**Interface layout:**

- **Ліва панель (40% екрану):** Список компонентів для вибору (MVP):
  - Банери
    - Варіант 1 (thumbnail preview)
    - Варіант 2
    - Варіант 3
    - ...
  - Таббар
    - Варіант 1
    - Варіант 2
    - ...
  - Хедер
    - Варіант 1
    - Варіант 2
    - ...
  - Сабнейли
    - Варіант 1
    - Варіант 2
    - ...
  - Сайдбар
    - Варіант 1
    - Варіант 2
    - ...
  - Тема
    - Світла
    - Темна
- **Права панель (60% екрану):**
  - Live Preview: Мобільний шаблон казино
  - Відображає:
    - Обраний концепт на банерах
    - Обраний неймінг на хедері
    - Обрані компоненти в реальному часі

**Interaction:**

- Клік на варіант компонента → preview оновлюється
- Radio buttons для вибору (один варіант на компонент)
- Можливість повернутись та змінити

**Output saved:** Component selections (banner type, tabbar type, header type, thumbnails type, sidebar type, theme)

### Step 9: Review & Submit

**Interface:**

- Summary всіх вибраних опцій
- Фінальний preview
- Можливість повернутись до будь-якого кроку

**Actions:**

- "Save as Draft" – зберегти для редагування пізніше
- "Save for Approval" – лінка на превʼю для ствердження з CPO/CEO
- "Approval" – інформація зберігається з можливістю завантажити PDF, або у вигляді презентаціїної сторінки з можливістю Share по лінці

**After Submit (потребує обговорення):**

- Slack нотифікації всім командам
- (Опціонально) Jira епіки створюються
